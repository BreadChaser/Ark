ARK reliability audit

15 July 2026 — focused on recurring state, session-switching, scrolling, long-chat, and validation failures.

What changed

False “working” state: status parsing now examines only the recent tmux screen rather than old scrollback, so historical “Working” text cannot keep a completed chat active.
No dropped just-sent messages: authoritative Codex transcript writes now reread messages inside their mutation lane. A capture that started before a browser send can no longer overwrite the new user message with an old snapshot.
Completed-session cache reset: starting a new Codex task clears the prior rollout and transcript cache, preventing old ready state or an old rollout path from leaking into the next task. Deleted sessions also clear their caches.
Chat switching and scrolling: an in-flight paged history request cannot overwrite a newer live capture, the live SSE stream is no longer reopened on every render, and a resize only follows the bottom when the reader was already there.
Less background churn: completed Codex sessions use their persisted final state instead of probing each tmux pane every three seconds. Ordinary capture ticks also stop rewriting session metadata when nothing changed.
A working release check: the browser smoke test had stopped halfway through on an invalid reference to private module state. It now checks the visible image-viewer DOM, and npm run verify runs both static/core checks and the full browser smoke suite.

Evidence

npm run verify passed on the audited tree. It includes syntax checks, parser fixtures, core/self checks, service-unit generation, and the isolated Firefox browser smoke suite.

Important remaining work

Long-chat transport is still the largest performance risk. Each active capture currently reads, merges, signs, and SSE-sends the full transcript. The UI only renders a tail, but it still receives and processes the whole history. The next substantial reliability slice should stream state plus a message tail/delta, with the existing paged message endpoint responsible for older history.
Remote terminal logs are copied in full. Remote pipe-log synchronization currently replaces the complete growing log each capture. It should use an offset-based append protocol, like the Codex transcript reader.
Message paging is UI pagination only. The server currently parses the entire JSONL file before selecting a page. A backward tail reader is worthwhile after capture payloads stop carrying full histories.
Shutdown remains abrupt. The systemd unit correctly keeps tmux alive, but the Node service should still close SSE/terminal attachments cleanly on SIGTERM. This is a small separate hardening task.

Suggested order

  1. Convert active chat capture to tail/delta transport and measure long-chat latency.
  2. Make remote pipe-log sync incremental.
  3. Add graceful Node shutdown and then profile memory by separating the Node process from preserved Codex/tmux workers.

This audit intentionally avoids redesigning ARK or changing its private-network deployment model.
