Ark is not reliable yet.

The central SSH/tmux architecture is viable, but the current working tree has data-loss paths, broken recovery controls, parser false positives, polling races, and UI code that has accumulated contradictory override layers.

Top findings

The most important problem is history ownership.

Ark says it stores full, clean history, but each capture rebuilds and rewrites that history from a lossy terminal snapshot. That is the opposite of the documented contract and can permanently discard messages.

Critical

Chat history is permanently truncated and deduplicated incorrectly

The terminal parser keeps only 160 lines and 80 parsed messages. The merge then keeps only 120 messages, deduplicating repeated messages solely by role and text. Every chat capture writes that reduced snapshot back to messages.jsonl, so older and intentionally repeated messages are lost.

server.mjs:1298, server.mjs:1384-1396, server.mjs:1788-1824; conflicts with ARK_PRODUCT_PLAN.md:143-149.

Critical

Capture and send requests can overwrite each other's files

Polling runs every 1.8 seconds without an in-flight guard. Capture and send both perform read-modify-write cycles on the same session and message files without a per-session queue or lock. A capture that read old messages can finish after a send and overwrite the newly stored user message.

hub/static/app.js:978-997, server.mjs:254-268, server.mjs:1224-1253, server.mjs:1360-1371.

Critical

Restart and Resume do not implement what their buttons say

The endpoint calls startTmux, which deliberately does nothing when the named tmux session already exists. If tmux is gone, session.runner_command takes precedence over the generated resume command, so Resume usually launches a fresh Codex command instead of resume --last.

server.mjs:341-354, server.mjs:956-967, server.mjs:998-1006; buttons at hub/static/index.html:197-198.

High

Slow captures can display the wrong session

A capture remembers the session at request start but stores its result in one global lastCapture without checking that the session is still active. Switching sessions while SSH is slow can render the previous session's terminal or chat under the newly selected session header.

hub/static/app.js:989-997, hub/static/app.js:1021-1068.

High

The parser deletes legitimate Codex answers

A whole message is rejected if it contains common project paths such as server.mjs or hub/static/, tool-like words such as Ran, or code fragments matched by broad regular expressions. Other filters remove any line mentioning a GPT model, any warning, and ordinary replies beginning with words such as Model, Directory, Working, or Thinking. These are normal contents of coding answers.

server.mjs:1399-1412, server.mjs:1983-1985, server.mjs:2011-2058.

High

The native model picker can send a different option than it labels

If menu parsing fails, Ark invents four hard-coded model choices and numeric positions. The parser only recognizes options 1 through 4. Codex menus and available models can change by account or release, so a button labelled with one model can send a number that selects another.

server.mjs:1835-1869, server.mjs:1918-1923. This is especially risky when testing newer Codex/model menus.

High

A failed send destroys the visible draft and attachment queue

The composer clears its text and queued files before awaiting the API. There is no catch/restore path. If the request fails, the draft is gone and its pending chat bubble remains in client state.

hub/static/app.js:1153-1178.

High

Private-network access is the only authorization boundary

Ark binds to every interface and accepts all loopback, RFC1918, link-local, and Tailscale-range source addresses. Any device on an accepted network can reach APIs that send terminal input, execute configured tools, manage sessions, and manage account material. This is a documented v1 choice, but it is a deployment blocker on any LAN or tailnet containing untrusted devices.

server.mjs:29-30, server.mjs:53-60, server.mjs:1748-1759.

Medium

Polling cost grows with the full tmux history

Every 1.8 seconds Ark captures up to 100,000 tmux lines, transfers them over SSH, parses them, and may rewrite history files. Long-running sessions will become progressively more expensive and mobile latency will amplify overlapping requests.

hub/static/app.js:978-982, server.mjs:1034-1041.

Medium

"Auto profile" is first-match selection, not availability routing

Routing chooses the first enabled profile whose executable exists. Authentication, rate limits, start failures, max_concurrent, and the configured routing strategy do not influence selection. A profile that explicitly needs login is still reported as available when the executable exists.

server.mjs:598-639, server.mjs:1456-1484, server.mjs:1578-1602.

Medium

Corrupt readable files fail open as empty data

Store, secrets, and message readers catch all parse/read failures and return empty collections. The next normal write can then replace recoverable data with an empty or partial file. Readable storage is a good choice, but corruption must stop writes and surface a repair error.

server.mjs:758-770, server.mjs:1239-1253, server.mjs:1351-1371.

Medium

Device discovery creates confusing duplicates and aliases

SSH config entries and Tailscale peers are inserted under unrelated IDs and are never reconciled by hostname, IP, or SSH target. One physical machine can appear twice, and non-machine SSH aliases can appear as devices. Slug collisions silently keep only the first entry.

server.mjs:397-422, server.mjs:443-487, server.mjs:490-524.

Medium

The GUI smoke report self-certifies many features

The diagnostics endpoint returns a long hard-coded list of true feature flags, while the smoke test asserts those same flags. This proves that the names exist, not that restart, routing, history preservation, mobile layout, or parser behavior works.

server.mjs:829-890, scripts/gui-smoke.mjs:253-297.

Medium

Documentation describes incompatible runtime shapes

The README says no package install is required even though startup imports four npm dependencies. The direction document says the agent runs on the selected repo's device, while remote chat currently runs Codex on the hub and asks it by prompt to use SSH. This makes deployment and debugging harder than necessary.

README.md:47-56, README.md:72-88, docs/DIRECTION.md:285-303, server.mjs:155-188.

Shortest repair order

  1. Stop capture from writing messages.jsonl. Make sends and completed assistant messages append-only; never reconstruct durable history from the screen.
  2. Serialize per-session server mutations and add a capture generation/in-flight guard in the client.
  3. Implement honest Restart and Resume semantics, including the exact Codex thread ID when available.
  4. Replace whole-message junk rejection with narrow line/chrome removal. Keep uncertain content instead of deleting it.
  5. Remove hard-coded model fallback choices. Only render controls parsed from the live menu; otherwise show a clear unsupported state.
  6. Restore drafts and queued attachments on send failure.
  7. Change capture from full-history polling to an incremental cursor or rely on the existing live PTY/log path.
  8. Make profile selection honor authentication and current availability before adding cost/usage balancing.
  9. Merge SSH and Tailscale device identities with an explicit alias record.
  10. After behavior is stable, consolidate the CSS and replace self-reported feature assertions with behavioral checks.
